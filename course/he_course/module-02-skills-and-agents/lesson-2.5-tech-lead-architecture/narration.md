# שיעור 2.5 (ElevenLabs / עברית)
---

## 00:00 – 00:50

בשיעור הקודם יצרנו PRD — מסמך שאומר **מה** בונים. עכשיו הסוכן הבא בתור: **Tech Lead** — הוא אומר **איך**.

מצרפים `@docs/prd.md` כקלט. זה מקור האמת. בלי זה הסוכן ינחש, ואנחנו כבר יודעים שניחושים עולים ביוקר.

## 00:50 – 02:10

אני כותב את הפרומפט ומפעיל. הסוכן לוקח את ה-PRD ומתחיל להחליט: באיזה סטאק נשתמש, איך מחלקים תיקיות, ואיפה הגבולות בין חלקי המערכת.

המטרה היא לא הטכנולוגיה הכי חדשה — אלא עקביות. אם הסוכן בוחר מבנה מסוים, כל הקוד שייכתב מעכשיו אמור להתאים לו.

כאן אפשר לראות את ההחלטות: React לפרונט, SQLite למקומי, מבנה תיקיות פשוט — routes, services, repositories. שימו לב שהסוכן גם בוחר **מה לא** לבנות בגרסה ראשונה.

## 02:10 – 03:30

הסוכן שומר את הפלט לקובץ `docs/architecture.md`. גם כאן — קובץ בפרויקט, לא הודעה בצ'אט.

פותחים ובודקים: סטאק, data model, מבנה תיקיות, גבולות API, ופרוסות מימוש — כלומר סדר הבנייה. קודם סכמה, אחר כך service, אחר כך route, אחר כך UI.

הקובץ הזה הולך איתנו לשיעורים הבאים. סוכן ה-UX, סוכן הפיתוח — כולם יצרפו אותו עם `@docs/architecture.md`.

## 03:30 – 04:45

לפני שממשיכים, אני שולח שאלה אחת: "מה חסר כדי להתחיל את הפרוסה הראשונה?"

הסוכן מחזיר רשימה קונקרטית: אילו קבצים צריך ליצור, אילו חבילות להתקין, ואיך בודקים שהחלק הראשון עובד.

אם התשובה כללית מדי — מהדקים: "תן שמות קבצים ספציפיים וסדר יישום."

## 04:45 – 05:45

לסיכום: לקחנו את ה-PRD, העברנו אותו דרך סוכן הארכיטקטורה, וסיימנו עם `docs/architecture.md`.

בשיעור הבא, 2.6, עוברים מהמבנה הטכני לזרימות המשתמש — מסכים, מצבים, ומה קורה כשמשהו נכשל. מצרפים גם `@docs/prd.md` וגם `@docs/architecture.md`.
